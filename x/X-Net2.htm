X-Frame Multi-Drop Serial Network (X-Net2)

Description

X-Net2, the X-Frame multi-drop serial network protocol introduced with Release 1.3 of the X-Frame software, is a protocol that provides external serial command and read-back of certain controls appearing in a MediaMatrix view file. The devices that may be controlled are, Level, SubPreset, Ramp, Switch, and certain Routers. Up to 256 controls in each X-Frame, selected by the designer from the above list, can be controlled through X-Net2. It is also possible to use X-Net2 to read the status of certain controls in an X-Frame. The devices that may be read are: Gates, the gate outputs of the Automatic Mixers, and SubPresets. The hardware required for this is a standard 2 wire, half-duplex, RS-485 serial port. The serial command protocol used by X-Net2 is human-readable ASCII. It is possible to test and debug X-Net control of a specific view file using just a PC with a terminal program, a spare serial port, and an appropriate RS-485 adapter. An X-Frame will not respond to any RS-485 commands unless there is a running view file in the X-Frame.

X-Net2 Devices

X-Net2 control is enabled by using control-wiring to wire an X-Net2 external control device to the device to be controlled. Note that there are two types of X-Net external serial control input devices: Level and Trigger. The X-Net2 Level devices may be connected to single or multi-channel Level devices, Ramp, Switch, or externally controllable Router devices. The X-Net2 Trigger devices may only be connected to SubPreset control-bus inputs, and are built into all SubPreset devices. The X-Net2 external serial output devices may be connected to the control-bus outputs of SubPreset, Gate, or Automatic Mixer devices.

Communications Protocol

The X-Net2 communicates at a fixed setting of 19200 Baud, 8-bit data, no parity, and 1 stop bit (19200,8,n,1). The X-Net2 supports both point-to-point and multi-drop configurations.

Message Protocol

Messages are ASCII (text) strings. While they contain hexadecimal numbers, the Hex numbers are represented by ASCII (text). Do not send actual hex data, but an ASCII (text) string that represents the value to be sent.

Each message begins with a message-type character and ends (with the exception of the NAK message) with an end-of-message character '.'. The message-type characters and the End Of Message (EOM) '.' character have been chosen so that they will not conflict with any characters that are possible in the body of the message. Each message to the X-Frame results in a response message from the X-Frame, with the exception of multi-drop broadcast messages.

In general, the protocol supports three main paradigms. These are the setting and getting of control-input values, getting control-output values, and sending trigger commands. User IDs, which identify a unique control, are specified with a 3 digit hexadecimal number, 000 through FFF. Note that the current version of X-Frame only supports up to 256 (FF Hex) devices, so the first digit of the User ID must be 0. In other words the usable range of user IDs is 000 to 0FF. Note that there is a separate numbering space for the User IDs in each control category, i.e. Trigger number 3 'T003' refers to a different X-Frame device than Set Value number 3 'S003', or Output number 3 'O003'. Control values are specified with a 2 digit hexadecimal number, 00 through FF.

The setting and getting of control-input values, using the cSETVALUE 'S' and cGETVALUE 'G' commands, are used to set the position or value for things such as a Level device’s Trim control or a Router device’s Select control. The 256 values that a control can take on correspond to 256 equidistant positions that a knob in an X-Frame view file connected to that control can take on. For example in a point to point system, to set the level control wired to Serial Input 14 in the view file to fully counter clockwise, one first must convert 14 which is a decimal number in the view file into Hex, which is 00E (remember that each user ID must be a 3 digit Hex number). Fully counter clockwise is 00 Hex. The cSETVALUE command is 'S' and the end-of-message character is '.'. Putting this all together gives a command string:

S00E00.

Note that it is not required or desirable to put a carriage return after the end-of-message (EOM) character '.'. The X-Frame will accept the command when it receives the EOM character. Shortly thereafter it will return the value that the control has been set to:

V00E00.

You must wait for this response before sending any other commands. This is required since we are using a 2 wire, half duplex protocol, and messages can only go one way at a time.

To check what a control is set to, you can use the cGETVALUE 'G' command. For example to check the setting of the level control wired to Serial Input 12, you send:

G00C.

Note that since you are requesting a control value, you do not send one. If the control was set to the mid point of its rotation you will get a response:

V00C80.

It is possible, and normal in some cases, for the value that a control actually takes on (and returns) to not match the value to which it was set. For instance, in the case of a switch-style control, any value sent between 00 and 7F Hex will result in a return value of 00 Hex, and any value sent between 80 and FF Hex will result in a return value of FF Hex. For example you may have sent to a switch wired to Serial Input 7:

S0073E.

However since the switch can only be on or off it will be set to off and return:

V00700.

If commanding a Level device’s Trim control, the dB range of the Trim control (set up by Trim Min. and Trim Max.) is simply divided into 256 possible values if it is not greater than 20 dB. If it is greater than 20 dB, the top 20 dB will be expanded to encompass the majority of the control range, and the rest of the dB range will be compressed into the lower portion of the control range. In this way the action of a conventional audio taper level control will be simulated.

In Appendix A are tables that map control values to Router channel selections for various sized Routers.

The output-control protocol, using cOUTPUTVALUE 'O', is used to check the value of devices such as gates. For a gate, the value will be either 00 or FF. For example to check the status of the gate wired to Serial Output 10 you would send:

O00A.

Note that you did not send a value, since this command is to check a value. If the Gate is open you will see this as the response:

V00AFF.

Or if the gate was closed:

V00A00.

The trigger-control commands, using cTRIGGER 'T', are used to load SubPresets. There are 2 basic types of SubPreset devices available in the Device menu: front panel or serial triggered; and front panel, BoB, or serial triggered. The front panel and BoB triggered SubPresets are all in the range of 1 to 8 since there are only 8 front panel buttons, and 8 BoB control inputs. There is also a User definable SubPreset which can be set internally to be any number from 1 to 100. If set to 1 through 8 they will also be front panel controlled, and be functionally the same as the other front panel SubPreset devices. Inside the User definable SubPreset device the number must be entered three times, once in the SubPreset itself, once in the Button device, and once in the Serial Trigger device. To trigger SubPreset number 1 you would send:

T001.

Again note you do not set a value since a trigger just commands a SubPreset to load. After the SubPreset has loaded (the delay is dependent on the number of controls in the SubPreset), you will get a response:

V001FF.

Note that this response could come back as a result of a set value cSETVALUE 'S', an output value cOUTPUTVALUE 'O', or a trigger cTRIGGER 'T' command. In each case it would be returning the current value of a different control in X-Frame. It is not possible to just look at the returned value and know what it is referring to without knowing the command that caused the response. For example you can set the value of the control wired to Serial Input 1:

S001FF.

or you can read the status of the gate wired to Serial Output 1:

G001.

or you can trip SubPreset 1:

T001.

and in each case get the same response back:

V001FF.

which would each refer to a different control in X-Frame. You can have in the same view file Serial Inputs, Serial Outputs, and Triggers all of which have the same number, but are totally independent of each other.

If you don’t want the user to be able to recall a SubPreset from the front panel, use the User SubPreset set to a number from 9 to 100. This is also suggested if you are using the multi-drop capability to recall SubPresets on more than one unit at a time. The front panel buttons will normally only control the SubPresets in the unit that they are on, and not any of the other units.

If you wish to allow the front panel buttons on one X-Frame to recall the SubPresets on multiple units, you need only wire a Serial Broadcast device to the output of the SubPreset device on the unit you wish to act as the master. When the SubPreset is triggered on this unit, whether by front panel button, virtual button on the X-Frame software screen, serial command, or BoB control input, a broadcast message will be transmitted from this master X-Frame to all the others on the network directing them to trigger the same SubPreset. The Serial Broadcast device determines which trigger command to transmit based on which SubPreset it is wired to. There will be a slight delay since the SubPreset must complete loading on the master unit before the broadcast message is sent. Only place the Serial Broadcast device(s) in one unit you select to be the master. If Serial Broadcast devices are used in more than one unit, you run the danger of creating a feedback loop, so that the first unit triggers the second and the second triggers the first.

The main use for the Serial Broadcast device is intended to be systems with multiple X-Frames but no control system such as an AMX or Crestron. If recalling SubPresets is all you need to do, this will save the cost of the control system.

If you are using this feature, it is best for the control system (if used) to listen for an unused RS-485 network before transmitting.

The get version command cGETVERSION 'P' is used to determine the revision number of the code stored in the flash memory of the X-Frame. To get the version you send:

P.

To which the X-Frame will respond with something like this:

Q020B.

This indicates that the major revision number is 02 Hex or 2 in decimal, and the minor revision number is 0B Hex or 11 in decimal. This corresponds to a front panel flash version number 2.11.

If you send a command using a Uid that is not in the view file, the X-Frame will respond with a cUNLISTEDUID 'U' and repeat back the Uid it does not find in the view file. For example if there was no Serial Input 37 in the view file (25 Hex), and you sent:

S02548.

the response would be:

U025.

This can happen easily if you forget that the decimal input and output numbers in the view file must be translated into Hex for use in X-Net2 control.

If you send an illegal message such as specifying a value as part of a get message, the X-Frame will respond with a Negative Acknowledge cNAK '?' instead of the normal response. For example if you sent:

G03057.

the X-Frame will respond with:

?

Note this is the only message in the X-Net2 protocol which does not end in an end of message cEOM '.'.

No response will be sent if garbled data or illegal commands are received by an X-Frame. These can be caused by collisions between messages from the control system, and responses from the X-Frame. Although no response is sent, the X-Frame does display X-Net2 errors on its front panel LCD such as: syntax (illegal commands), framing (reversed RS-485 wires, hardware problems), and overflow (too many broadcast commands).

Point to Point Operation

In a point to point configuration, commands are identical to the former X-Net, and the responses are also identical except for the response to the cGETVERSION 'P' command. Every command sent will result in a response back from the X-Frame. Since the serial connection is half-duplex, the control system must wait for the X-Frame response to be received before attempting to send another command. The exact delay will vary randomly, so a fixed delay before sending another command, unless extremely long, can’t be used reliably. It is particularly easy to get into trouble with the next command sent after triggering a SubPreset, since the acknowledgement is not sent back until after the SubPreset is finished loading. The time this takes is determined by the number of controls to be set in a given SubPreset, plus a random variation. If the control system attempts to send a second command before the X-Frame has finished sending its response, the two messages may collide, and the X-Frame will not respond correctly. Point to point commands are NOT prepended with the character constant cMULTIDROP 'M' and the multi-drop address. The response message from the X-Frame does NOT include its multi-drop address (even if set in the view file). Point to point messages sent on a multi-drop network will cause all the units to react, and send response messages. Since these response messages will not all come back at the same instant in time, it will be very difficult for the control system to determine when it is safe to send a second command. There could be contention, which may cause a hardware failure! For this reason it is not suggested to attempt sending point to point messages in a multi-drop system. Instead use the broadcast capability of the multi-drop addressing if you wish to send the same message to all units.

Multi-drop Addressing

In a multi-drop configuration, messages can be routed to a specific X-Frame by prepending the message with the character constant cMULTIDROP 'M' and two hexadecimal characters indicating the multi-drop address of the X-Frame. Messages can be broadcast to all X-Frames on a multi-drop network by using the broadcast address FF Hex. Note that the X-Frame does NOT return a response to a broadcast message. The default X-Frame multi-drop address is 0, which is the NULL address. The X-Frame does NOT respond to any multi-drop messages while its multi-drop address is equal to 0. The X-Frame WILL send a response to all other multi-drop messages. The address can be changed through the ‘Compile Options’ dialog box under the ‘File’ menu. The X-Frame unit multi-drop address as entered into the ‘Compile Options’ dialog box can be any decimal value from 0 to 254. The corresponding hexadecimal value of 00 to FE must be used in the X-Net2 command. For example to tell unit 10 to set Serial Input 14 fully counterclockwise you would send:

M0AS00E00.

and the X-Frame would respond:

V00E00.

Note that the response does not include the unit address.

The X-Frame also has the capability to generate broadcast multi-drop trigger messages using the Serial Broadcast device (new in Release 1.3). This device may only be wired to the control output of a SubPreset device. When the SubPreset is triggered by the front panel push button, BoB control input, or serial command, and has completed loading, it will trigger the Serial Broadcast device which will relay the trigger command to all other X-Frames on the network whose multi-drop addresses are other than 0. If they have a corresponding SubPreset, it will be triggered. This capability was designed to allow a single front panel push button or BoB control input to be used to trigger SubPresets in multiple X-Frames even in cases where X-Net2 is not otherwise being used.

Electrical Connections

Note that RS-485 can only support 31 X-Frames without the use of repeaters. For further information on RS-485 communication we suggest reading B&B Electronics "RS-422 and RS-485 Application Note" available from the B&B web site:

http://www.bb-elec.com/

X-Net2 is 2 wire RS-485. The RS-485 connections on the rear of the X-Frame are labeled "Net". There is no difference between the two sets of Net connections, they are simply tied together to facilitate daisy-chaining of multiple X-Frames. The "-" connection is equivalent to the RS-485 defined "A" connection, and the "+" connection is equivalent to the RS-485 defined "B" connection. There is a ground connection available next to the Net connections (it is not labeled), which should be used. Even though this type of circuit is called "2 wire", it is important to provide a third wire ground connection for reliable operation. A good cable to use for RS-485 is common and inexpensive Cat 5 computer network cable. Use one pair for the RS-485 signal and the other pairs as ground connections. X-Frame does not use isolated RS-485 connections, and does not bias or terminate the connections. When long RS-485 cables are used, placing a 120 ohm termination resistor across each of the two extreme ends of the line will help performance. No more than two terminators should ever be used on an RS-485 circuit.

In cases where the local grounds are of different potential, or in noisy electrical environments, it is suggested that optical RS-485 isolators be used. Besides improving noise immunity and increasing data reliability in extreme environments, they also can help protect the X-Frame’s RS-485 port from damage due to electrical surges.

C-LIKE MESSAGE DEFINITION

This section defines the message protocol using "C-like" declarations of constants and structures. The convention here is that words beginning with ‘c’, such as cSETVALUE and cGETVALUE, are character constants. Words beginning with ‘f’’, such as fUid and fVal, are message field structures. Words beginning with ‘m’, such as mSetValue and mGetValue, are message structures.

//---- Message constants

const char cGETVERSION

= 'P';

// an fType value

const char cMULTIDROP

= 'M';

// an fType value

const char cSETVALUE

= 'S';

// an fType value

const char cGETVALUE

= 'G';

// an fType value

const char cOUTPUTVALUE

= 'O';

// an fType value

const char cTRIGGER

= 'T';

// an fType value

const char cVALUE

= 'V';

// an fType value

const char cUNLISTEDUID

= 'U';

// an fType value

const char cVERSION

= 'Q';

// an fType value

const char cNAK

= '?';

// an fType value

const char cEOM

= '.';

// an fEom value


//---- Message fields
/* fType: Message Type. 1 character, denotes the message type.
*/
struct fType {
char data;
};

/* fAddr: Multi-drop Address. 2 ASCII hexadecimal digits, specifies the X-Frame’s address on the multi-drop X-Net Serial network.
*/
struct fAddr {
char data[2];
};

/* fUid: User ID. 3 ASCII hexadecimal digits, the first of which must be a 0. Specifies the Id in the X-Net External Serial device. Note that each category of devices has its own set of Id’s to choose from, so that X-Net Level Input number 4 is different from X-Net Output number 4, etc.
*/
struct fUid {
char data[3];
};

/* fVal: Control Value. 2 ASCII hexadecimal digits, specifies a control value between 0 and 255.
*/
struct fVal {
char data[2];
};

/* fVer: Version number. 4 ASCII hexadecimal digits, specifies a major version number between 0 and 255 and a minor version number between 0 and 255.
*/
struct fVer {
char data[4];
};

/* fEom: End Of Message. 1 character, appears at the end of every message, excepting mNak.
*/
struct fEom {
char data = cEOM;
};

//---- Messages

/* mSetValue: Set Control Value. Request sent by a client to set the value of a Serial Input device. */
struct mSetValue {
fType type = cSETVALUE;
fUid id;
fVal val;
fEom eom;
};

/* mGetValue: Get Control Value. Request sent by a client requesting the value of a Serial Input device.
*/
struct mGetValue {
fType type = cGETVALUE;
fUid id;
fEom eom;
};

/* mGetOutput: Get Output Value. Request sent by a client requesting the value of a Serial Output device.
*/
struct mGetOutput {
fType type = cOUTPUTVALUE;
fUid id;
fEom eom;
};

/* mTrigger: Set Trigger Value. Request sent by a client to send a trigger event.
*/
struct mTrigger {
fType type = cTRIGGER;
fUid id;
fEom eom;
};

/* mGetVersion: Get Software Version. Request sent by a client requesting the software version.
*/
struct mGetVersion {
fType type = cGETVERSION;
fEom eom;
};

/* mValue: Control Value. Response sent to the client acknowledging an mGetValue, mGetOutput, mTrigger, or mSetValue. Note that it is possible, and normal in some cases, that the val field will not match the val that was sent in the mSetValue. Note that for mTrigger messages, the response will always be ‘FF’, and for mGetOutput messages, the response will be ‘00’ if the output is not set, and ‘FF’ if the output is set.
*/
struct mValue {
fType type = cVALUE;
fUid id;
fVal val;
fEom eom;
};

/* mUnlistedUid: Unlisted User ID error. Response indicating the fUid specified in the mSetValue or mGetValue was not found in the X-Frame.
*/
struct mUnlistedUid {
fType type = cUNLISTEDUID;
fUid id;
fEom eom;
};

/* mVersion: Software Version. Response indicating the software version running in the X-Frame.
*/
struct mVersion {
fType type = cVERSION;
fVer ver;
fEom eom;
};

/* mNak: Negative Acknowledge. Response sent to client upon receipt of illegal message (i.e. specifying data for a ‘Get’ message. Note that no response is sent for garbled data or illegal command codes. The only message that does not end in fEom!
*/
struct mNak {
fType = cNAK;
};

 

Flow Control

X-Net2 is implemented using a half-duplex RS-485 protocol, with the X-Frame acting as a slave. Therefore, except when the Serial Broadcast device is used, the X-Frame will never send out a character unless asked to, and in most cases the master must wait for a response before sending the next message. Also, the X-Frame will not send a NAK unless the message was completely delivered but just not right (i.e. a value specified for an mGetOutput or mTrigger message, or a missing value for a mSetValue message).

Serial Control Example and Experimentation

Using a terminal program you can experiment with X-Net2 serial control. Your terminal program must be set for 19,200 baud, 8-bit data, no parity, and 1 stop bit (19200,8,n,1), and for a serial port that is not otherwise in use on your computer. You also must either have a serial output card capable of 2 wire RS-485 communications, or use an external converter such as a B & B Electronics 485PTBR. (http://www.bb-elec.com/)

The 485PTBR like most RS-485 cards and adapters has separate transmit and receive terminals. This is to allow 4 wire 485 operation. For two wire operation the TD(A) terminal must be connected to the RD(A) terminal and to the wire going to one of the - Net terminals on the back of the X-Frame. The TD(B) terminal must be connected to the RD(B) terminal and to the wire going to one of the + Net terminals on the back of the X-Frame. A wire must also be run from one of the SG terminals on the 485PTBR to one of the unlabeled ground terminals next to the Net terminals on the rear of the X-Frame. Solder an 110,000 ohm resistor at point R2 on the 485PTBR. This sets the receive/transmit switching circuit for 19,200 baud operation. Plus 12 VDC must be connected to the +12VDC terminal of the 485PTBR from a power supply such as the 485PS2, and the negative lead connected to one of the SG terminals.

Plug the 485PTBR into the RS-232 connection on your computer that was selected in your terminal program, and you should be ready to experiment.

For example, since the number 14 in decimal is E in Hex, and the number 47 in decimal is 2F in Hex, then to set the value of the 14th Serial Input device to 47, type:

S00E2F.

The response should be:

V00E2F.

and the position of both the knob in the Serial Input device and the device it is wired to should change. To get this value later, type:

G00E.

The response should be:

V00E2F.

To load the 7th preset, type:

T007.

The X-Frame will respond with:

V007FF.

To check the status of the 6th Serial Output device, type:

O006.

and the response should be either:

V00600.

Or

V006FF.

If a device is unknown, the response will be:

U###.

Where ### is the user ID, instead of:

V###vv.

The LCD of the X-Frame will display any errors received on the X-Net2, either syntax or protocol.

Backwards Compatibility with X-Net

X-Net2 is backwards compatible with X-Net. The differences are:

1. X-Net did not support multi-drop operation.

2. When you sent a cGETVERSION 'P' command you did not need the cEOM '.' to follow it. Therefore what you would send would just be:

P

3. The response you got to this command would look like:

X-Frame Xternal Network: Ver 2.9

4. Level controls in X-Net were strictly linear in dB, and were not modified beyond a 20 dB range.

5. Version 1.2 and earlier of X-Frame do not contain the User SubPreset, Serial Broadcast, or remote controllable 1 by N Router devices introduced in version 1.3, and are not compatible with them.
